# Web App Implementation Guide

## Objective
Build a web page on top of the existing Python codebase by:
1. Adding a lightweight API layer.
2. Building a frontend that calls the API.
3. Reusing existing `src/` logic directly.
4. Documenting the technical reasoning, process, and tradeoffs.

## Why the API Layer Is Needed
The existing codebase is Python runtime logic plus CLI entrypoints (`python -m src.main ...`). A browser cannot import or execute this server-side Python directly.

An API layer is the boundary that:
- Executes Python logic on the server.
- Exposes safe JSON/HTTP interfaces for the browser.
- Prevents frontend duplication/reimplementation of core behavior.

Without an API layer, options are limited:
- Static-generation only: no interactive live routing/search.
- Full logic rewrite in JavaScript: duplicated behavior and drift risk.
- Pyodide/WASM Python in browser: heavier complexity and compatibility constraints.

## Architecture
The implementation adds:
- Backend: `FastAPI` app in `src/web_app.py`
- Frontend: static assets in `web/`
  - `web/index.html`
  - `web/static/styles.css`
  - `web/static/app.js`

The FastAPI app serves both:
- API endpoints under `/api/*`
- frontend at `/` and static assets under `/static/*`

## Reuse of Existing Logic (No Reimplementation)
All endpoint behavior delegates to existing modules:
- `build_port_manifest()` from `src/port_manifest.py`
- `QueryEnginePort(...).render_summary()` from `src/query_engine.py`
- `get_commands(...)` from `src/commands.py`
- `get_tools(...)` from `src/tools.py`
- `PortRuntime().route_prompt(...)` from `src/runtime.py`
- `ToolPermissionContext` from `src/permissions.py`

No new business logic model was created in the frontend.

## API Endpoints Added
- `GET /api/summary`
  - Returns: summary markdown generated by `QueryEnginePort`.
- `GET /api/manifest`
  - Returns: manifest metadata and top-level modules.
- `GET /api/commands`
  - Query params: `limit`, `query`, `include_plugin_commands`, `include_skill_commands`
  - Returns: filtered mirrored command entries.
- `GET /api/tools`
  - Query params: `limit`, `query`, `simple_mode`, `include_mcp`, `deny_tool`, `deny_prefix`
  - Returns: filtered mirrored tool entries.
- `GET /api/route`
  - Query params: `prompt`, `limit`
  - Returns: routed command/tool matches for the prompt.

## Frontend Design
The page is intentionally visual and “demo-ready” while still minimal:
- Hero metrics (Python files, commands, tools).
- Prompt router panel with live match rendering.
- Searchable command/tool panes.
- Runtime summary pane.
- Animated gradient background, glass panels, staged reveal animation.
- Responsive layout for desktop and mobile.

Stack:
- Plain HTML/CSS/JavaScript (no framework runtime required).

## How to Run
From repo root:

```bash
python -m uvicorn src.web_app:app --host 127.0.0.1 --port 8000 --reload
```

Then open:
- `http://127.0.0.1:8000/`
- API docs: `http://127.0.0.1:8000/docs`

## Consequences and Alternatives if API Is Not Used
### Consequences
- No live server-side routing/search from existing Python runtime.
- Frontend must either become static or duplicate logic.
- Harder to keep behavior aligned with CLI/runtime.

### Alternatives
1. Static snapshot site
- Precompute outputs with CLI.
- Serve static HTML/JSON.
- Best for docs, not interactive runtime demos.

2. Full JS port of runtime features
- Frontend directly computes route/search/summary.
- High maintenance and drift against Python source.

3. Pyodide in-browser execution
- Runs Python in browser, no traditional backend.
- Larger payload, slower startup, module compatibility caveats.

For this project goal (interactive minimal clone + direct reuse of existing Python logic), the lightweight API layer is the most practical design.
